Hermes配Agnes全链路调试

微信消息协议铁律定型 · 内在探索工作室上线
🔥 三条最佳经验
1Hermes custom provider 的 key 必须放 OPENAI_API_KEY
配 Hermes 主模型为 Agnes（agnes-2.0-flash）时，按历史成功经验写 .env 的 model.api_key 字段，但重启报 401。读 cli.py:3838-3844 源码发现——custom provider 只读 `OPENAI_API_KEY` 或 `OPENROUTER_API_KEY` 环境变量，完全不读 `model.api_key`。
→ 可复用：配 custom provider 时 key 放 OPENAI_API_KEY，.env 的 model.* 写 base_url + provider + default。源码才是真相
2微信纯文本通道不能加粗——iLink 协议级铁证
今早微信消息 `**\u201c一棵树...` 暴露问题。查 iLink 协议：`item_list[].type` 只有 1文本/2图片/3语音/4文件/5视频，无富文本型。OpenClaw 官方源码有 `markdownToPlainText()` 把 markdown 转纯文本——官方认证无粗体。7 个自动化 prompt 全线修复。
→ 可复用：微信是纯文本通道，** 不渲染粗体。真·粗体唯一路径是图片方案（待主人拍板）
3内在探索咨询工作室 6 Skill 从设计方案一次性蒸馏完成
基于设计方案文档一次性创建：`inner-explore-team`（团队管理器）+ 5 角色 Skill（罗杰斯/克里希那穆提/德波顿/卢森堡/克利尔）。团队管理器让主人不用记人名，说场景自动匹配。
→ 可复用：设计方案成熟后一次蒸馏成 Skill 很高效；团队管理器做入口，角色各自独立
⚠️ 做得不好的事
1Hermes 配 Agnes 耗时过长——源码入手应更早
第一次 401 后没有直接读源码的 API key 注入逻辑，而是在 .env 字段名上试错。实际读了 cli.py 第 3838-3844 行就真相大白：custom provider 只读 OPENAI_API_KEY。历史写法（simple_switch.py 的 model.api_key）对该版本 Hermes 主程序无效。
→ 改进：custom provider 的 key 不确定时，第一动作读 provider 源码的 env var 读取逻辑，而非查历史写法
